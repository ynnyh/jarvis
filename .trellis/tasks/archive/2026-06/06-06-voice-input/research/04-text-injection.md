# Research: 文字注入聚焦输入框（剪贴板+粘贴 vs 模拟键入）

- **Query**: enigo 模拟键入 vs 剪贴板+Ctrl+V；Windows 下中文/IME 兼容性坑；剪贴板保存恢复
- **Scope**: external + 内部（现有窗口/命令模式）
- **Date**: 2026-06-07

## 结论先行（v1 推荐）

**v1 推荐「剪贴板 + 模拟 Ctrl+V」方案，并做剪贴板的「保存→写入→粘贴→恢复」以避免污染用户剪贴板。**

理由：直接模拟键入（enigo `text()`）在 **Windows 上对中文/Unicode 的支持不可靠**——它走的是按键事件，遇到中文/表情/特殊字符经常丢字或乱码，且会与系统 IME 状态打架。而「把文字塞进剪贴板再模拟 Ctrl+V」是绕开 IME、对任意 Unicode 都稳的工业级做法（很多输入工具都这么干）。代价是会动用户剪贴板，用「用完即恢复」补偿。

## Findings

### 两套方案对比

| 维度 | enigo 直接模拟键入 `text()` | 剪贴板 + 模拟 Ctrl+V（推荐） |
|---|---|---|
| 中文/Unicode | **Windows 上不稳**，易丢字/乱码，受 IME 干扰 | 稳，任意 Unicode 都行 |
| 实现复杂度 | 看似简单，实则要处理逐字符与布局 | 中等（要管剪贴板存取 + 一次 Ctrl+V 组合键） |
| 副作用 | 不污染剪贴板 | 污染剪贴板 → 用「保存+恢复」补偿 |
| 对目标程序兼容性 | 取决于程序如何收键 | 几乎所有能接受粘贴的输入框都行 |
| 焦点要求 | 需目标窗口聚焦 | 同样需目标窗口聚焦 |

### 涉及的 crate（当前仓库都没有，需新增）

- **`enigo`**（已核实最新 `0.6.1`）：跨平台输入模拟。无论哪套方案都要用它来**发 Ctrl+V 组合键**（或方案一的逐字符 `text()`）。enigo 0.5→0.6 有 API 变化（builder/`Settings`），落地以拉到的版本为准。
- **剪贴板**两条路：
  1. **`arboard`** crate（跨平台剪贴板，独立于 Tauri）——直接读写文本，控制力强，便于「保存旧值→写新值→粘贴→恢复旧值」。
  2. **`tauri-plugin-clipboard-manager`**（已核实 `2.3.2`，Tauri 官方插件）——与前端打通方便，但前端读写、要走插件权限，做「精确恢复」不如 arboard 直接。
  - **v1 建议在 Rust 端用 `arboard`**：因为整段流程（采音→转写→注入）都在 Rust 后端做，注入也在后端一气呵成最顺，不必绕前端。

### 推荐的注入流程（剪贴板方案，避免污染）

```
1. 读取并暂存当前剪贴板内容（文本，可能为空/可能是图片→图片场景下只能尽力，文本优先）
2. 把转写结果写入剪贴板
3. 模拟按下 Ctrl+V（enigo）→ 触发目标输入框粘贴
4. 等极短延时（确保目标程序读完剪贴板，经验 ~50-150ms）
5. 把第 1 步暂存的内容写回剪贴板（恢复）
```

注意点：
- **焦点时序**：模拟 Ctrl+V 前，焦点必须在用户原来的输入框上。Jarvis 的小人窗口是 `alwaysOnTop` + `skipTaskbar` + 可能 `focus`——录音/转写期间**不能抢走目标窗口焦点**，否则 V 粘到自己窗口里了。这与小人窗口的焦点管理强相关（见下「与本仓库的结合」）。
- **恢复时序竞态**：第 5 步恢复太早会把还没粘完的内容冲掉；太晚用户可能已经手动复制了别的。150ms 量级是经验折中，需实测。
- **剪贴板是图片/富文本时**：arboard 读文本会失败，恢复只能尽力（v1 可只保证文本场景的完美恢复，非文本场景接受「被覆盖」并在文档里说明）。

### Windows IME / 中文的具体坑

- 直接 `enigo.text("你好")` 在 Windows 上不是走 IME 组字，而是尝试逐 Unicode 注入，常见结果是丢字、顺序错乱、或只出部分字符 —— 这是放弃方案一的核心原因。
- Ctrl+V 粘贴**绕过 IME**，文字以剪贴板内容整体进入，不经过组字流程，所以中文稳。
- 模拟 Ctrl+V 本身在 Windows 上要发 `Ctrl down → V down → V up → Ctrl up` 的正确序列，enigo 的组合键 API 处理这个；注意有些程序对 `Ctrl+V` 与 `Shift+Insert` 响应不同，Ctrl+V 通用性最好。

### 与本仓库的结合（现成可复用）

- **焦点管理**：本仓库窗口逻辑（`src-tauri/src/commands/window.rs`、`lib.rs` 的 avatar 窗口 setup）大量处理 show/hide/focus。注入场景需要「小人窗口**不夺焦**」——可参考 `tauri.conf.json` 里各窗口的 `focus`/`skipTaskbar`/`alwaysOnTop` 配置，录音触发的窗口或交互要避免 `set_focus()`。
- **后端一条龙**：采音、转写、注入都在 Rust，参考现有 `#[tauri::command] async fn` 模式（`commands/` 目录）暴露一个 `voice_input_start` / `voice_input_stop` 之类命令即可，无需前端介入文字注入。
- **防黑窗**：若 plan B 走 sidecar，注意复用 `silent_command`（`commands/mod.rs`，`CREATE_NO_WINDOW`）。

## Caveats / Not Found

- arboard 确切最新版本号本环境查询被截断，未确认；`cargo add arboard` 取最新即可。
- 未实测「Ctrl+V 后多久恢复剪贴板」在目标程序上的安全延时，需开发时实测调参。
- 不同目标程序（微信、浏览器、IDE、Office）对粘贴的响应一致性未逐一验证；Ctrl+V 是最通用选择，但极个别程序可能需要特殊处理，属边缘情况。
- 是否需要「粘贴 vs 直接替换选区」等高级行为不在 v1 范围。
