# 轻量语音输入工具（语音转写 + 注入聚焦框，非系统输入法）

> 状态：**存根（planning）**。2026-06-06 记下想法 + 拍板范围，待正式 brainstorm 细化后再开工。
> 当前优先级低于进行中的 `06-04-ui-theme-redesign`。

## Goal / 价值

随时按热键（或点桌面小人）说话 → 实时转写 → 把文字直接塞进**当前聚焦的输入框**。
核心卖点：**本地 + 轻量 + 隐私**——市面语音输入（讯飞那类）都重、强依赖云、隐私存疑，
而 Jarvis「本地优先的个人助手」定位恰好能补这个缺口，也是用户当前急缺的能力。

## 范围决策（2026-06-06 已拍板）

**做「工具型」，不做真·系统输入法（IME）。**

- ✅ 工具型：全局热键 / 点小人 → 录音 → 语音转文字 → 注入聚焦输入框（剪贴板 + 模拟 Ctrl+V，或模拟键入）。
  轻量、可基于现有桌面小人 + Tauri 落地，是真正缺的东西。
- ❌ 真·IME：接系统输入法框架（Windows TSF / macOS IMKit），系统级、巨重、维护成本极高，
  与「轻量」诉求相悖——**明确不碰**。

## 待 brainstorm 决策点（Open Questions）

- **STT 引擎**：本地（whisper.cpp / sherpa-onnx，最贴本地优先，但模型体积大、需评估首包体积与性能）
  vs 云端（快、好接，但要联网 + key，弱化本地优先卖点）。← 最关键的一道分叉。
- **文字注入机制**：剪贴板 + 模拟 Ctrl+V（简单、但会污染剪贴板）vs 直接模拟键入（不污染、但中文/IME 兼容性需验证）。跨平台输入模拟方案（Windows API / enigo 等）待定。
- **触发方式**：全局快捷键、点桌面小人、或两者都要；录音中/转写中的状态如何用小人反馈。
- **转写形态**：录完再转（整段）vs 流式实时上屏；是否要自动标点、热词、中英混输。
- **范围**：先只做中文，还是中英混输 / 多语言。

## 现有可复用点（待 brainstorm 核实，勿当定论）

- 桌面小人（cow pet）——天然的触发入口 + 录音/转写状态可视化载体。
- Tauri 全局快捷键、窗口/穿透能力（参考现有窗口系统经验）。
- 本地优先 / 密钥走 OS keychain 的既有理念（若选云端 STT，key 照此处理）。

## Out of Scope（暂定）

- 不做系统级输入法（IME）。
- 不做语音命令 / 语音控制 Jarvis（那是另一条线，本任务只管「语音 → 文字 → 注入」）。
