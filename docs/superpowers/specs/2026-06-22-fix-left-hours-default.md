# 修复剩余工时（left）正确递减

**日期**：2026-06-23
**分支**：建议新开 `fix/left-decrement`（不要沿用 `fix/fine-report-viewlet-path`，两者无关）
**状态**：待用户审核

> **方向变更说明**：本 spec 初稿（2026-06-22）原本讨论"固定写 1"，经用户反思后否决——
> 不应为个人习惯拆掉原有保护，应让 `left` 实现禅道字段的正确语义（按本次工时递减）。
> 本文档已整体重写。

## 背景

用户反馈"禅道里剩余工时（left）数值不对"。

经查 `src-tauri/src/zentao.rs:668-786` 的 `record_workhour`，问题根因是 **`left[1]={current_left}`（原值回填）导致 left 永远不递减**：
- 写之前 GET 任务读 `current_left`（L668-681）
- 写时 `left[1]={current_left}`（L698），**只回填原值，不减本次工时**
- 缺 `left` 时中止报错（防误关）

结果：无论写多少工时，禅道里的 `left` 永远保持初始值不变——这正是"数值不对"的根因。

## 目标

实现 `left` 字段的正确语义：**写 X 小时工时，left 相应减 X**，并避免误触发禅道"left=0 → 任务自动关闭"的行为。

## 决策（已与用户确认）

| 场景 | 处理 |
|---|---|
| 旧 `left` > 0 且正常 | 新 `left = max(0.5, 旧 left − 本次工时)` |
| 旧 `left` = 0 / 负数 / 字段缺失 | **中止报错**（带优化提示） |
| 计算结果 ≤ 0.5 | **卡在 0.5**，不让禅道自动关任务 |

**为什么卡 0.5 而不是 0**：禅道在 `left=0` 时会把任务自动标记为完成，这会掩盖"任务实际未完成"的真实状态。卡 0.5 保留递减语义、又不误触发关闭。

**为什么不默认 left=1**：用户明确否决——"不应该为自己的习惯买单"。基准必须是真实的旧 left，发现旧值异常应中止而不是猜。

## 方案

### 后端改动（`src-tauri/src/zentao.rs`）

**`record_workhour` 方法（L658-786）**：

1. **保留** `current_left` 读取（L668-681）——这是递减基准，必须读
2. **优化"缺 left"报错文案**（L676-681）：
   - 旧文案强调"防误关"
   - 新文案改为："任务 #X 当前 left（剩余工时）为 {0/负/缺失}，无法作为递减基准。请在禅道确认该任务状态（可能已完成/已关闭），或手动修正 left 后重试。"
3. **新增递减计算**（在 L694 form body 构造前）：
   ```rust
   // left 按本次工时递减，但卡在 0.5 避免禅道自动关任务
   let new_left = (current_left - hours).max(0.5);
   ```
4. **修改 form body**（L698）：
   - `format!("left[1]={}", current_left)` → `format!("left[1]={}", new_left)`
5. **修改 `EffortResult` 构造**（L781）：
   - `preserved_left: current_left` → 改为返回**递减后的新值**，字段语义从"保留的旧 left"变成"写入的新 left"
6. **字段重命名考虑**（L794 `EffortResult.preserved_left`）：
   - 选项 A（推荐）：**字段名保留** `preserved_left`，加注释说明语义已变，避免破坏 MCP 工具序列化契约
   - 选项 B：重命名为 `new_left`，需同步改 `effort_logging.rs:91,115` 的 JSON key
   - **本 spec 选 A**，理由：MCP 工具的返回值 JSON key 是外部契约，重命名会让旧调用方读不到字段

### MCP 工具返回值（`src-tauri/src/tools/effort_logging.rs`）

**无需改动**：`preservedLeft: r.preserved_left`（L91, L115）现在返回的是递减后的新 left，字段名不变，消费端不受影响。

### 前端改动

**无**。理由：
- 前端紧迫度评分（`stores/app.ts:148-156`）用的 `leftHours` 来自**禅道任务的实时读取**，不是写入时的返回值——left 正确递减后，评分自然就准了
- RiskWindow 的工时预算/超载分析同样基于任务的实时 `left`，会自动反映正确的递减

> 这正是"合理计算差值"相比"固定写 1"的最大优势——前端零改动，紧迫度系统天然恢复准确。

## 测试

### 后端单元测试（`zentao.rs` 或对应 test 文件）

新增/更新以下用例（建议覆盖在 test 模块里）：

| 用例 | 输入 | 预期 |
|---|---|---|
| 正常递减 | left=10, hours=3 | new_left=7, 写入成功 |
| 递减后卡 0.5 | left=2, hours=5 | new_left=0.5（不报错），写入成功 |
| 递减刚好等于 0.5 | left=3, hours=2.5 | new_left=0.5，写入成功 |
| 刚好等于 0.5 边界 | left=2.5, hours=2 | new_left=0.5，写入成功 |
| 旧 left=0 | left=0, hours=1 | **中止报错**（优化文案） |
| 旧 left 为负 | left=-1, hours=1 | **中止报错** |
| 旧 left 缺失 | task 无 left 字段 | **中止报错**（沿用现有逻辑） |

### 手动验证

1. 用 MCP 工具 `log-task-effort` 给一个 left=10 的任务写 3h
2. 在禅道确认：`consumed` +3、`left` 变成 7
3. 再写 8h：禅道里 `left` 卡在 0.5、任务**未**被自动关闭
4. 找一个 left=0 的任务写工时：返回优化后的报错文案

## 风险

| 风险 | 缓解 |
|---|---|
| 卡 0.5 阈值不适应所有团队（有人希望真能减到 0） | YAGNI，先按用户确认的 0.5 实现；未来需要时再加配置项 |
| `preserved_left` 字段名与"递减后的新值"语义不符 | 加注释说明；如未来要重命名，同步改 `effort_logging.rs` 两处 |
| 旧 left 异常时报错可能让批量写工时中断 | 这是正确行为——批量场景本就该在中止时让用户决策 |
